# Common Data Model API Platform Requirements

> 作成日: 2026-05-26
> Status: Requirements draft v3 (baseline)

## 1. 目的・ミッション

BigQuery 上の基幹システムデータをもとに、**標準データモデル（共通データモデル）** を構築し、社内の各ビジネスユニットが **REST API として利用できる状態にする**。直近のミッションは API 提供の実現。

## 2. データモデル

### 2.1 提供範囲（段階的）

| フェーズ | 内容 | 例 |
|---|---|---|
| 第1段階 | 基礎リソース単体 | 注文情報、会計伝票情報 |
| 第2段階以降 | 複数リソースを統合した、ややビジネスロジックを含むモデル | 未払注文＋計上済注文をユニオン＋重複排除した「注文統合ビュー」 |

### 2.2 ガバナンス

- 共通データモデルの所有・更新責任は **中央チーム**。
- 命名規約・バージョニング規約・上流（基幹）スキーマ変更追従ポリシーは **本プロジェクトで定義**。

### 2.3 未確定（設計判断ポイント）

- **複雑な統合ロジック（ユニオン＋重複排除など）の配置先**：BigQueryビュー / マテリアライズドビュー / 中間テーブル / API層 のどれにするか。

## 3. アーキテクチャ前提

### 3.1 データ層

- データソース: GCP / BigQuery（既存）。
- データ鮮度向上は別ワークストリーム、本件スコープ外。

### 3.2 API実行層

- 社内 VPC 内の Azure 上で稼働。
- **既設の Azure API Management** をフロントに利用（**既存共有インスタンス**）。
  - マルチテナント運用観点で、命名規約・サブスクリプション・製品（Product）構成のルール整理が必要。
- バックエンド実行基盤は社内で閉域利用可能な Azure サービス（Functions / App Service / Container Apps 等）から選定。
- 将来 **iPaaS 導入予定** → APIM・本サービス・iPaaS の役割分担を意識した設計とする。

### 3.3 ネットワーク構成

- 社外向け通信は **Proxy 経由のみ**。Azure → BigQuery も Proxy 経由。
  - ⚠ **Proxy 仕様（認証方式・対応プロトコル）が現時点で不明** → 設計フェーズ初期で確認が必要。BigQuery クライアントの Proxy 対応可否に影響。
- Azure ↔ オンプレ SQL Server は **ExpressRoute** 経由で接続可能。
- GCP 側のネットワーク制約は特になし。

### 3.4 オンプレ依存

- ユーザー属性は **オンプレ SQL Server に既に実装・運用されている**。
  - 設計初期で **既存スキーマ調査** を実施し、APIで参照する属性インターフェースを定義する。

## 4. 認証・認可

### 4.1 認証

- ID 提供: **Entra ID（OAuth 2.0 / OIDC）**。
- 利用者: 人間ユーザー / サービスアカウント の両方。
- 認証方式は **モダン** を志向（OIDC + JWT、サービスアカウントは Client Credentials または Managed Identity を想定）。

### 4.2 認可属性の取得

- Entra ID から取得した ID をキーに、**オンプレ SQL Server** から部門・役職などの属性を取得して認可判定。
- **サービスアカウント側の属性管理方式は追って検討**。

### 4.3 行・列レベル制御

- 行レベル: 起票者単位 〜 事業単位の複数粒度に対応する想定。
- 列レベル: 原価・単価などのセンシティブ項目を制御対象として想定。
- **実装場所はサービス層（API バックエンド）を第一候補**。BigQuery 側の RLS/CLS は補助的位置づけ。

## 5. API 設計方針

- **OpenAPI 3.1 準拠**。
- リソース指向の RESTful 設計。クエリ・ページング・ソート等は **業界デファクトに準拠**（具体規約は設計時に選定）。
- バージョニング戦略を初期から定義（URI path vs Header）。
- レスポンスは JSON。

## 6. 性能・規模

### 6.1 目標

- **API レスポンス目標は、エンドユーザー体感として 100〜500ms**。
- BigQuery 単体クエリは秒オーダーになり得るため、以下の二段構えで SLO を整理する：
  - **キャッシュヒット時**：100〜500ms を目標値とする。
  - **キャッシュミス時**：許容上限を別途定義（例: 数秒以内）。
- 利用規模は初期は小規模、段階的に拡大。

### 6.2 キャッシュ

- キャッシュ層の導入は **許容**。
- 配置先はモダンな選択肢を踏まえて **案を提示する**（APIM 組込み / Azure Cache for Redis / BigQuery BI Engine / マテビュー / アプリ内 など）。

## 7. 監査・運用観点

- 監査ログ（誰が・いつ・何を・どの条件で参照したか）は **必須**。
- ログ出力・監視基盤：
  - 第一選択: **Azure Log Analytics**（クラウド純正）。
  - 可能であれば **Datadog** も併用。
- 個人情報・データレジデンシー要件は現時点なし（将来要件への拡張余地を持たせる）。

## 8. 開発体制・スタック方針

- 実装言語・FW は未指定。選定指針：
  - **Claude / Devin による AI 支援開発** が機能しやすい、ドキュメント・コミュニティ・型情報が成熟したスタック。
  - **OpenAPI 3.1 / OAuth2 / Azure SDK / BigQuery クライアント** の対応が成熟している言語。
  - 候補（後で比較）: Python (FastAPI) / TypeScript (NestJS, Hono) / C# (.NET minimal API) 等。
- **AI 支援開発のためのコーディング規約・リポジトリ標準（AGENTS.md / CLAUDE.md 等）の整備は別プロジェクトで対応**。

## 9. 設計フェーズで判断・調査が必要な論点（未確定リスト）

| # | 論点 | 概要 |
|---|------|------|
| L1 | 複雑ロジックの配置 | BigQuery ビュー / マテビュー / 中間テーブル / API 層 の比較 |
| L2 | キャッシュ配置 | APIM 組込み / Redis / BI Engine / マテビュー / アプリ内 の比較 |
| L3 | 行・列レベル制御の実装方式 | 属性ベース (ABAC) / OPA 等のポリシーエンジン / 独自実装 |
| L4 | API バックエンドのランタイム | Functions / App Service / Container Apps / AKS の比較 |
| L5 | API 設計規約 | JSON:API / OData / 独自、ページング・フィルタ表現の選択 |
| L6 | OpenAPI 連携方式 | コードファースト vs スキーマファースト |
| L7 | サービスアカウントの属性管理 | SQL Server 同居 / Entra ID アプリクレーム / 別レジストリ |
| L8 | iPaaS 将来導入を見据えた境界設計 | APIM / 本サービス / iPaaS の役割分担 |
| L9 | Proxy 仕様の調査 | BigQuery クライアントが社内 Proxy を通せるかの実機確認 |
| L10 | 既存 SQL Server スキーマ調査 | 認可属性として参照するテーブル・カラムの確定 |

## 10. 次のステップ案

1. **既存資産の調査リスト化** … Proxy 仕様、SQL Server スキーマ、APIM の現行運用ルールなど、設計に進む前に社内ヒアリング・調査すべき項目をチェックリスト化。
2. **アーキテクチャ案の作成** … 上記論点を踏まえ、2〜3 パターンのアーキテクチャ全体図と各案のトレードオフを比較。
3. **技術スタック比較** … API バックエンドの言語・FW・ランタイムを、AI 支援開発との相性も含めて比較。
4. **L1〜L10 個別深掘り** … 気になる論点から先に調査・案出し。

推奨順序: **1 → 2 → 3**（前提を確定させてからアーキテクチャ案に進むことで手戻りを最小化）。
